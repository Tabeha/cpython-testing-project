Виды тестов

В проекте использованы несколько уровней проверки

Unit tests

Проверяют одну функцию или небольшой блок логики.

Regression tests

Подтверждают, что старая ошибка не вернулась после изменений.

Integration tests

Проверяют, как несколько частей системы работают вместе.

Performance tests

Измеряют время выполнения и помогают сравнить подходы.

В этом проекте основной акцент сделан на unit и performance тесты, потому что их проще всего показать на компактном учебном примере и при этом получить измеримый результат.
Сравнение фреймворков

`unittest` и `pytest` решают одну задачу, но делают это по-разному

`unittest`

  • входит в стандартную библиотеку Python
  • использует `TestCase` и методы `assertEqual`, `assertRaises`
  • подходит для классического xUnit-стиля
  • удобен там, где важен стандартный встроенный инструмент

`pytest`

  • использует обычный `assert`
  • дает более короткий и читаемый код тестов
  • удобен для параметризации и разбора падений
  • часто быстрее в повседневной разработке
Вывод: оба инструмента рабочие. Для учебного и прикладного кода `pytest` обычно удобнее, а `unittest` полезен как стандартная встроенная база.
Практическая часть

Тесты стали тяжелее, чем простая проверка `2 + 2`

  • базовые тесты всех арифметических операций
  • проверка деления на ноль
  • набор фиксированных кейсов
  • stress-test на 2000 детерминированных входов
Это делает сравнение двух фреймворков содержательным: они тестируют один и тот же код не только на простых случаях, но и на более широкой выборке данных.

Статический анализ

Тестов недостаточно без линтера и тайпчекера

`ruff`

Проверяет стиль, потенциальные ошибки и проблемные конструкции в коде.

ruff check .

`mypy`

Проверяет, насколько код согласован с аннотациями типов.

mypy .
Тесты проверяют поведение, а `ruff` и `mypy` добавляют раннюю проверку структуры и типовой корректности кода.

CPython

В CPython тестирование — это уже полноценная инфраструктура

1

Локальные проверки до PR

Разработчик запускает локальные тесты и утилиты вроде `make patchcheck`.

2

GitHub checks после открытия PR

Изменения должны пройти автоматические проверки до merge.

3

Buildbots на разных платформах

Они ловят проблемы, которые не видны на одной локальной машине.

4

Проверки после merge

Часть ошибок проявляется уже после commit в основной репозиторий.

Главная мысль: CPython показывает, что тестирование — это не один запуск `pytest`, а система из локальных тестов, CI, многоплатформенных buildbots и пост-merge мониторинга.
Лабораторная часть

Плохие тесты тоже полезны, если использовать их как учебный материал

Нет `assert`

Тест формально запускается, но ничего не проверяет.

Неверное ожидаемое значение

Показывает прямое падение при неправильной логике проверки.

Слабая проверка

Условие слишком широкое и не гарантирует корректность.

Нет проверки исключения

Код падает, а тест не оформляет это как ожидаемое поведение.

Shared mutable state

Тесты начинают зависеть друг от друга.

Copy-paste ошибка

Частый практический источник ложных падений и путаницы.
